# LandscapeAI RAG 功能设计文档

> 版本：V1.0 | 日期：2026-07-09 | 阶段：V1.1 RAG 升级

## 1. 背景与目标

### 1.1 当前 MVP 方案
- 知识库：7 个指标 × 15 条规范条文，约 690 tokens
- 方式：全量注入——每次审查把全部条文塞进 Prompt
- 局限：知识库扩大后 Token 成本线性增长，AI 注意力分散

### 1.2 RAG 升级目标
- 知识库扩展到 10 本规范、100-150 条条文（全量注入 ~8000 tokens，接近模型上限）
- 通过语义检索只注入相关条文（Top-25，~2000 tokens），节省 75% Token
- 让 AI 注意力聚焦于相关规范，提升审查准确率
- 前端可视化展示检索过程，让 RAG "可见可验证"

## 2. 架构设计

### 2.1 整体流程

```
用户方案描述
    ↓
[Embedding] 描述 → 384 维向量
    ↓
[检索] 余弦相似度 vs 150 条向量 → Top-25 条文
    ↓
[Prompt 组装] System Prompt + 检索到的 25 条 + User Prompt
    ↓
[AI 审查] agnes-2.0-flash → 结构化 JSON 结果
    ↓
[前端展示] 审查结果 + 参考规范面板（检索来源）
```

### 2.2 离线准备阶段（启动时执行一次）
1. 加载全部 150 条规范条文
2. 用 `Xenova/multilingual-e5-small` 模型向量化每条条文
3. 缓存到 `embeddings.json`（~230KB）
4. 后续启动直接加载缓存

### 2.3 在线检索阶段（每次审查请求）
1. 将用户方案描述向量化（同模型）
2. 计算与 150 条条文的余弦相似度
3. 取 Top-25 条最相关条文
4. 按指标分组，确保 7 个指标都有覆盖

## 3. 技术选型

| 组件 | 选型 | 理由 |
|------|------|------|
| Embedding 模型 | `Xenova/multilingual-e5-small` | 384 维，支持中文，模型仅 90MB |
| 运行框架 | `@xenova/transformers` (Node.js) | 纯 JS + WASM，无需 Python |
| 向量存储 | 内存 + JSON 文件缓存 | 150 条 × 384 维 ≈ 230KB，无需数据库 |
| 相似度算法 | 余弦相似度（Cosine Similarity） | 标准方案，归一化后等同于点积 |
| 检索策略 | 全局 Top-K（K=25） | 简单有效，按指标分组展示 |

## 5. Prompt 变更

### 5.1 MVP 方式（全量注入）
```
##规范知识库
以下是住宅项目相关的规范条文，请严格对照审查：
[全部 150 条条文，~8000 tokens]
```

### 5.2 RAG 方式（检索注入）
```
##规范知识库
以下是通过语义检索找到的与本方案最相关的 25 条规范条文，请严格对照审查：
[检索到的 25 条条文，~2000 tokens]
注：如某指标未检索到相关条文，请返回 status=信息不足
```

## 6. 前端改造

### 6.1 加载页分阶段提示
- 阶段 1："正在检索相关规范..."（检索耗时 <100ms）
- 阶段 2："正在 AI 审查..."（AI 调用 ~15s）

### 6.2 结果页新增"参考规范"面板
- 折叠式面板，位于审查结果下方
- 展示检索命中的条文：规范名称 + 条款号 + 相关度分数
- 按指标分组显示

## 8. 性能预期

| 指标 | MVP 全量注入 | RAG 方式 | 改善 |
|------|-------------|----------|------|
| Prompt Token | ~8000 | ~2000 | -75% |
| 检索耗时 | 0 | <100ms | 可忽略 |
| AI 响应时间 | ~15s | ~10-15s | 略快（输入更短） |
| 审查准确率 | 基准 | 预期提升 | 注意力更聚焦 |

## 9. 风险与应对

| 风险 | 应对 |
|------|------|
| transformers.js 兼容性问题 | 备选方案：Jina AI Embedding API |
| 模型下载慢 | 首次启动耐心等待，后续走缓存 |
| 检索准确率不足 | 调整 Top-K 值，或加入关键词预筛 |
| 知识库条文准确性 | 标注为草稿，由 12 年景观经验用户审核 |
